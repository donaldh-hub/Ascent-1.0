BUILD OBJECTIVE:

You are building a web-based application called **Ascent 1.0**.

Ascent 1.0 is NOT a simple task tracker.

It is a **continuous improvement operating system** designed to give users real-time visibility into workflows, risks, bottlenecks, assets, and execution performance.

The system must show:

* What is happening
* Why it is happening
* What it is costing
* What needs to happen next

This is a **cause → effect → action platform**, not a static database.

---

CORE PURPOSE:

The purpose of Ascent 1.0 is to help teams:

1. See operational health instantly using a RED / YELLOW / GREEN system
2. Track workflows step-by-step and identify bottlenecks
3. Capture real-world evidence (photos, documents, voice) inside workflows
4. Recalculate impact when changes occur (delays, approvals, change orders)
5. Track assets, warranties, and maintenance lifecycle
6. Surface risks, delays, and failures BEFORE they become critical
7. Drive users toward the NEXT BEST ACTION

This system is an **operational control tower**, not just a tracker. 

---

SYSTEM ARCHITECTURE (MANDATORY):

The platform must be built in THREE LAYERS:

1. INPUT LAYER
   This is how users feed the system:

   * Workflow updates
   * Status changes
   * Photo capture
   * Document uploads
   * Voice notes / voice-to-text
   * Asset installs and warranty inputs
   * Dates, reminders, follow-ups

2. LOGIC LAYER
   This is the intelligence engine:

   * Stoplight scoring engine
   * Flow calculator
   * Risk calculator
   * Improvement calculator
   * Execution calculator
   * Workflow health scoring
   * Operational health scoring
   * Alert & warning engine
   * Impact recalculation engine
   * Asset & warranty intelligence

3. OUTPUT LAYER
   This is what users SEE:

   * Main dashboard
   * Workflow dashboards
   * Alerts and notifications
   * Bottleneck visuals
   * Reports and trends
   * Asset & warranty views

RULE:
Input feeds Logic.
Logic feeds Visibility.
Visibility drives Action. 

---

CORE PRINCIPLE:

This system must ALWAYS answer:

* Are we on track?
* Where is the bottleneck?
* What is at risk?
* What changed?
* What should we do next?

If the UI cannot answer those in seconds, it is wrong.

SHARED CALCULATION ENGINE (CRITICAL):

ALL modules must feed into FOUR CORE SCORES:

1. FLOW SCORE
   Measures movement, delays, bottlenecks

2. RISK SCORE
   Measures severity, frequency, exposure

3. IMPROVEMENT SCORE
   Measures progress (Kaizen, DMAIC, fixes)

4. EXECUTION SCORE
   Measures follow-through and completion

These four scores combine into:

→ OPERATIONAL HEALTH SCORE (MAIN DASHBOARD)

Each score must output:

* Numeric value
* Stoplight color (Red / Yellow / Green)

DOCUMENT INTELLIGENCE (DIFFERENTIATOR):

Documents must NOT be stored blindly.

The system must:

* Detect document type
* Suggest where it belongs
* Link it to workflow / asset / step
* Allow manual correction
* Display documents INSIDE workflows

RULE:
Documents must live in CONTEXT, not folders. 

---

IMPACT RECALCULATION ENGINE (CRITICAL FEATURE):

When something changes (example: delay, approval, change order):

The system must:

1. Capture the event
2. Identify what changed
3. Recalculate timeline
4. Recalculate workflow score
5. Recalculate dashboard impact
6. Show what caused the change
7. Show cost of the change

RULE:
The system must show IMPACT, not just activity.

---

ASSET + WARRANTY INTELLIGENCE:

Each asset must track:

* Install date
* Warranty start & expiration
* Model & serial
* Maintenance schedule
* Life expectancy
* Service history

System must calculate:

* Days remaining on warranty
* Maintenance due
* Asset health

RULE:
Every asset becomes a TRACKED LIFECYCLE.
