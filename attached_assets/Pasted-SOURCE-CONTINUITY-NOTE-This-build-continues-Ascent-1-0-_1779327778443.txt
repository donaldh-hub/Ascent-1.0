BUILD NAME

Build 7.2.1 — Turn / Work Order Reporting Mode Assessment

BUILD PURPOSE

Build 7.2.1 adds organization-level reporting configuration that tells Ascent how to interpret the relationship between turns and work orders before running reporting analysis.

This is necessary because organizations track turns differently.

Some organizations keep turns and work orders separate.

Other organizations use unit-level work orders as the operational proof of turn progress.

Ascent must ask the organization how it tracks turns and then adjust reporting language, analysis grouping, Control Tower interpretation, and supporting record logic accordingly.

This build prevents Ascent from misclassifying turn-related work orders or mixing turn and work order language incorrectly.

CORE PRINCIPLE

Do not force one operational model onto every organization.

Ask the organization how they track turn progress.

Then report accordingly.

NEW ORGANIZATION SETTING REQUIRED

Add an organization-level setting called:

Turn / Work Order Reporting Mode

The setting should support these values:

1. Separate Turn and Work Order Tracking

Meaning:

The organization tracks turns as their own workflow or turn board.

Work orders are treated as normal service/maintenance demand unless explicitly linked to a turn.

Reporting behavior:

- Turn reports should use turn records.
- Work order reports should use work order records.
- Work order categories should not be treated as turn stages.
- Turn bottlenecks should come from turn stages.
- Work order backlog should not automatically become turn delay.
- If a work order is linked to a turn, show it as supporting context, not the main turn progress model.

2. Work Orders Measure Turn Progress

Meaning:

The organization uses unit-level work orders to track what is being completed during a turn.

Work order completion may be used as evidence of turn progress.

Reporting behavior:

- Work orders associated with vacant units, make-ready categories, turn labels, or turn-linked units may be interpreted as turn progress records.
- Unit-level work order closure can contribute to turn completion analysis.
- Delayed turn-related work orders can contribute to turn bottleneck analysis.
- Work order categories may support turn stage interpretation if mapped properly.
- Reports must clearly label these as “Turn-Related Work Orders” instead of ordinary reactive maintenance.
- Work order counts used for turn progress must remain traceable to supporting records.

3. Hybrid / Not Sure Yet

Meaning:

The organization may use both methods, or the user does not know yet.

Reporting behavior:

- Keep turn and work order analysis separate by default.
- Flag possible turn-related work orders as “Potential Turn-Related Work Orders.”
- Do not merge them into turn progress until confirmed.
- Surface a recommendation asking the user to confirm their tracking method.
- Use Qualified Analysis or Insufficient Data where appropriate.
- Do not create confident turn conclusions from ambiguous records.

IMPACT ON BUILD 7.2 ANALYSIS

Update Build 7.2 analysis behavior based on this setting.

MODE 1: Separate Turn and Work Order Tracking

Work Order Analysis:

- Treat work orders as maintenance/service demand.
- Do not interpret work order category “Turn” as a turn stage unless the record is explicitly linked to a turn.
- If “Turn” appears as a work order category, label it as an imported work order category.
- Do not mix those records into turn progress automatically.

Turn Analysis:

- Use turn records, make-ready records, turn stages, vacant dates, and turn status.
- Linked work orders may appear as supporting records only if there is a clear relationship.

Reports Language:

Use:

- Work Order Performance
- Work Order Time Allocation
- Turn Performance
- Turn / Make-Ready Time
- Linked Work Orders where applicable

Do not call ordinary work orders “turn progress.”

MODE 2: Work Orders Measure Turn Progress

Work Order Analysis:

- Identify turn-related work orders using available indicators, such as:
  - vacant unit status
  - make-ready labels
  - category equals Turn
  - description includes turn/make-ready context
  - linked turnId
  - unit currently in turn
  - status/stage mapped to turn process
  - organization-defined turn mapping if available

Turn Analysis:

- Allow turn-related work orders to contribute to turn progress.
- Use completed turn-related work orders as completion evidence.
- Use open/late turn-related work orders as turn delay evidence.
- Group turn-related work orders by unit/property.
- If stage mapping is available, map work order category/status to turn stage.
- If stage mapping is not available, label as “Turn-Related Work Orders — Unmapped Stage.”

Reports Language:

Use:

- Turn-Related Work Orders
- Work Orders Measuring Turn Progress
- Unit Turn Progress
- Turn Completion Evidence
- Turn Delay Evidence
- Unmapped Turn Work Orders

Do not count these only as ordinary reactive maintenance if the mode says work orders measure turn progress.

MODE 3: Hybrid / Not Sure Yet

Work Order Analysis:

- Keep work orders in work order reporting.
- Detect possible turn-related work orders but do not fully merge them.
- Label them as “Potential Turn-Related Work Orders.”

Turn Analysis:

- Use confirmed turn records for turn conclusions.
- Show potential turn-related work orders as a qualified insight only.
- Require user confirmation before treating them as turn progress.

Reports Language:

Use:

- Potential Turn-Related Work Orders
- Needs Confirmation
- Qualified Analysis
- Reporting Mode Not Confirmed

Do not generate confirmed turn progress from ambiguous work order records.

TURN-RELATED WORK ORDER DETECTION

Add reusable logic that identifies candidate turn-related work orders.

Candidate signals may include:

1. Work order category equals Turn, Make Ready, Make-Ready, Unit Turn, Turnover, Vacant, Prep, Punch, Paint, Flooring, Cleaning, Inspection, Rework

2. Work order description includes turn-related language

3. Unit status is vacant, notice, down, make-ready, or turning where available

4. Work order is attached to a turnId or workflowId representing a turn

5. Work order open date falls inside the unit’s turn/vacancy window

6. Work order status/stage aligns with known turn workflow stages

7. Property has organization configuration saying work orders measure turn progress

Detection output must include:

- isTurnRelatedCandidate
- turnRelationConfidence
- turnRelationReason
- linkedTurnId when available
- linkedUnitId
- linkedPropertyId
- stageMapping when available
- needsConfirmation boolean

Confidence states:

- Confirmed Turn-Related
- Likely Turn-Related
- Possible Turn-Related
- Not Turn-Related

Only Confirmed or Likely records may influence turn analysis in Work Orders Measure Turn Progress mode.

Possible records should remain qualified.

- Use labels like “Possible turn-related work orders need confirmation.”
- Do not let ambiguous records drive confirmed Turn Performance scoring.

LANGUAGE SEPARATION RULE

This build must strengthen the language separation rule.

The system should never accidentally mix terms.

If the data model says work order, call it a work order.

If the data model says turn, call it a turn.

If the organization uses work orders to measure turns, call them:

Turn-Related Work Orders

or

Unit-Turn Work Orders

Do not hide the record type.

This avoids false clarity.

REPORTING CONFIDENCE RULES

The selected mode must influence confidence.

Separate Mode:

- Native turn records can support Confirmed Turn Analysis.
- Work orders only support turn analysis if explicitly linked.
- Unlinked work orders cannot create confirmed turn conclusions.

Work Orders Measure Turn Progress Mode:

- Confirmed turn-related work orders may support Confirmed or Qualified Turn Analysis depending on completeness.
- Likely turn-related work orders may support Qualified Analysis.
- Possible turn-related work orders require confirmation.

Hybrid Mode:

- Native turn records may support confirmed turn analysis.
- Possible turn-related work orders stay qualified or separate.
- Ambiguous work orders must not create confirmed turn conclusions.
