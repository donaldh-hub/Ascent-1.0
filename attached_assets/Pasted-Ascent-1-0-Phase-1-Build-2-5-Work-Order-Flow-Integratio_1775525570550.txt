==================================================
CORE PRINCIPLE (NON-NEGOTIABLE)
===============================

ALL work order data must:

* produce real, clickable insights

If a work order cannot be traced → it must NOT generate a signal.

==================================================

1. WORK ORDER DATA INGESTION SYSTEM
   ==================================================

Create a Work Order Ingestion Service.

Must support:

* CSV upload (primary)
* structured manual entry (secondary)

Required fields (minimum):

* work_order_id
* property_name
* unit_number
* category (appliance, plumbing, etc.)
* description
* priority (if available)
* created_date
* first_response_date (if available)
* completed_date (if available)
* status

System must:

* normalize fields
* clean formatting
* pass through Assignment Engine (Build 1.7)
* attach to correct unit/property

==================================================
2. WORK ORDER → WORKFLOW TRANSLATION
====================================

Create a translation layer that converts work orders into system-recognized flow.

Two supported modes:

A. Attach to Existing Workflow

* if matching workflow exists (e.g., "Turns", "Work Orders")

B. Auto-Generate Workflow Items

* create workflow item under system-defined "Work Orders" workflow

Required default stages:

* Submitted
* Assigned
* In Progress
* Completed

Rules:

* created_date → Submitted
* first_response_date → Assigned
* active work → In Progress
* completed_date → Completed

All stage movement must:

* generate history
* feed timing engine

==================================================
3. RESPONSE TIME + SLA ENGINE
=============================

Build SLA tracking layer.

Track:

* time to first response
* total completion time

Default rule:

* 24-hour response expectation

Outputs:

* SLA met / SLA missed
* response delay duration

These MUST feed:

* scoring engine (risk + execution)
* alert engine (late response alerts)
* 1.8 signals (clickable SLA violations)

==================================================
4. CATEGORY INTELLIGENCE (FOUNDATION)
=====================================

Group work orders by category.

Examples:

* refrigerator
* plumbing
* electrical
* HVAC

System must:

* count frequency by category
* track repeat issues per unit

Outputs feed:

* risk scoring
* dashboard insights
* 1.8 signals

Example signal:

"40% of recent work orders are refrigerator-related"

MUST be clickable → show those work orders

==================================================
5. SIGNAL INTEGRATION (CRITICAL — TIES TO 1.8)
==============================================

ALL work order insights must be converted into SIGNALS compatible with Build 1.8.

Each signal must define:

* signal_type: "work_order"
* source: flow / risk / execution
* query_rules
* display_config

Examples:

A. SLA VIOLATION SIGNAL

* "12 work orders exceeded 24-hour response"
* CLICK → show those work orders

B. CATEGORY SPIKE SIGNAL

* "Refrigerator issues trending high"
* CLICK → filtered list

C. AGING WORK SIGNAL

* "8 work orders stuck in progress"
* CLICK → show items

RULE:

If it cannot produce a result set → it cannot exist as a signal.
