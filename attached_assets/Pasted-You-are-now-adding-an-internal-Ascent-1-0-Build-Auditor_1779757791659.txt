# REQUIRED AUDIT OUTPUT STRUCTURE

Every audit report must use this exact structure:

## 1. Build Audit Status

Status must be one of:

- PASS
- PARTIAL
- FAIL
- NEEDS MANUAL VERIFICATION

Include a short explanation.

## 2. Build Summary

Summarize what changed in plain English.

Include:

- New files
- Modified files
- Removed files, if any
- New routes
- Modified routes
- New components
- Modified components
- New data logic
- Modified data logic

## 3. Requirement Match

Compare the completed work against the build objective.

Classify each requirement as:

- Complete
- Partial
- Missing
- Not Verifiable

## 4. Wiring Validation

Check whether each visible UI element is connected to real behavior.

Validate:

- Buttons
- Upload actions
- Report generation actions
- Dashboard tiles
- Drilldowns
- Links
- Share/download actions
- Navigation routes
- Expand/collapse cards
- Filters
- Property selectors
- Record-level detail views

Each item must include:

- UI element name
- Expected behavior
- Actual discovered behavior
- Status
- Risk level

Risk level must be:

- Low
- Medium
- High
- Critical

## 5. Route Integrity

Inspect all relevant routes.

Identify:

- Working routes
- Missing routes
- Dead routes
- Routes that render blank content
- Routes that return Not Found
- Routes that display the wrong page
- Routes that are visually correct but not connected to state/data

## 6. Data Flow Verification

Check whether uploaded data, mock data, seeded data, or parsed report data flows correctly into the app.

Validate:

- Upload input
- Parsing
- Normalization
- Storage/state handling
- Dashboard metrics
- Drilldowns
- Recommendations
- Report output
- Download/share output

Mark anything that is hardcoded, mocked, or placeholder.

Do not allow hidden placeholder logic to pass as production-ready logic.

## 7. Ascent Product Promise Check

Confirm whether the build supports the Ascent 1.0 promise.

The build must help users move from:

Report upload → intelligence → operational conversation → better decision.

Check whether the feature helps answer:

- What is happening?
- Why does it matter?
- What records support it?
- What should the manager review first?
- What conversation should leadership have next?

If the build only creates visuals without decision-support value, mark it as Partial or Fail.

## 8. Screenshot / Manual Evidence Needed

Generate a checklist of screenshots or manual tests needed to confirm anything the system cannot verify automatically.

Each test must include:

- Test name
- Click path
- Expected result
- Evidence needed
- Pass/fail criteria

## 9. Problems Found

List all problems clearly.

For each problem include:

- Problem
- Location
- Why it matters
- Severity
- Recommended fix

Severity must be:

- Low
- Medium
- High
- Critical

## 10. Next Replit Prompt

Generate a clean, copy-paste-ready Replit prompt that fixes the highest-priority problems or moves the next build forward.

The prompt must be specific.

It must include:

- What to fix
- What not to break
- Required files/routes/components to inspect
- Required behavior
- Required validation
- Required final report from the agent

# REQUIRED WIRING VALIDATION RULES

The Wiring Validation Engine must be strict.

A feature is NOT complete unless:

1. The UI element exists.
2. The UI element is clickable or interactive where expected.
3. The action produces the correct result.
4. The result uses real connected data or clearly labeled mock/demo data.
5. The output connects to the correct dashboard, report, or record view.
6. The user can trace the insight back to supporting records.
7. No route produces a dead page, blank page, or unrelated page.
8. No critical button is decorative only.
9. No dashboard tile shows a metric that cannot be supported by data.
10. No recommendation appears without explaining what records or data triggered it.

If any of these fail, the status must be Partial or Fail.

# REQUIRED ASCENT-SPECIFIC VALIDATION AREAS

The auditor must pay special attention to these Ascent 1.0 areas:

## Operational Performance Tiles

Validate that these four tiles exist and are properly wired:

1. Work Order Performance
2. Turn Performance
3. PM Performance
4. Asset Performance

Each tile should support drilldown behavior.

Each tile should connect to supporting records or explain what records are missing.

## Work Order Intelligence

Validate:

- Work order count
- Open/closed status
- Response time
- Completion time
- Bottlenecks
- Category trends
- Preventable patterns
- Unit/property context
- Technician/team context, if available

## Turn Performance

Validate:

- Turn count
- Turn stage
- Percent complete
- Vacancy clock
- Stage bottlenecks
- Staff vs contractor responsibility, where available
- Vacancy cost exposure, where available
- Unit/property context

## Preventative Maintenance

Validate:

- PM logs
- PM gaps
- PM overdue signals
- Emergency work order relationship, if available
- Recommendations connected to PM findings

## Asset Performance

Validate:

- Asset inventory
- Install dates
- Warranty status
- Expired warranties
- Unit/property context
- Financial exposure
- Replacement/service risk
- Clickable affected asset records

## Recommendations Layer

Validate that recommendations are not vague commands.

Recommendations should be analysis-led.

They should explain:

1. What the analytics suggest
2. What to review first
3. Why it matters
4. What records support it
5. What operational question the manager should answer next

Do not allow recommendations like “fix this now” without context.
