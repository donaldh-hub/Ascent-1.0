BUILD NAME

PM Data Mapping Layer

PURPOSE OF THIS BUILD

Build 7.5 creates the preventative maintenance data mapping layer for Ascent 1.0 reporting.

This build does NOT create final PM scoring.

This build does NOT create PM recommendations yet.

This build does NOT create predictive PM conclusions.

This build creates the mapping contract that allows uploaded PM records to be normalized, classified, assigned, and prepared for future PM rules, PM gap detection, PM insight generation, and PM dashboard/reporting layers.

The purpose is simple:

Turn messy PM records into mapped, report-aware operational records that can be trusted, reviewed, filtered, and connected to supporting records.

5. PM Status Normalization

Normalize PM statuses into stable states.

Use:

- Completed
- Scheduled
- Due Soon
- Overdue
- Deferred
- Canceled
- Unknown

If a record has a completed date, it may be treated as Completed unless conflicting fields indicate otherwise.

If a record has a due date in the past and no completed date, it may be treated as Overdue.

If a record has a due date and no completed date, it may be Scheduled or Due Soon depending on existing date logic.

If status cannot be determined, classify as Unknown and add a warning.

Do not create fake completion.

Do not assume work was completed without evidence.

6. PM Date Logic

Add PM date parsing and normalization.

Support common date formats.

The mapper should calculate when possible:

- daysPastDue
- nextDueDate
- lastCompletedDate
- dueDateConfidence
- completedDateConfidence

If due date is missing, the record may still be mapped but must not be treated as fully reportable for due-date-based reporting.

If completed date is missing, the record may still be mapped but must not be treated as confirmed completed unless status clearly supports completion.

7. PM Reporting Eligibility

Classify each mapped PM record into the existing reporting eligibility states:

Fully Reportable

Use when the record has enough information to support PM reporting with confidence.

Minimum requirements:

- organizationId
- propertyId or confirmed property match
- PM task or PM category
- status or due/completion date logic
- enough date context to support the type of report being generated

Partially Reportable

Use when the record has some usable PM data but is missing key fields.

Examples:

- property matched but unit missing
- task exists but due date missing
- due date exists but status unclear
- category known but assignment unresolved
- completed date exists but asset/unit unclear

Not Reportable Yet

Use when the record cannot safely support PM reporting.

Examples:

- no property match
- no PM task/category
- no meaningful date/status information
- unresolved row that cannot be connected to the organization/property/unit context

Do not silently mix these states as equal truth.

8. PM Resolution State

Also classify each PM record using the existing resolution model:

- Fully Resolved
- Partially Resolved
- Unresolved

This is not the same as reporting eligibility.

A record can be partially resolved but still partially reportable.

A record can be unresolved and not reportable yet.

Preserve both concepts.

9. Mapping Confidence

Each PM record should receive a mapping confidence state.

Use the existing confidence pattern if one exists.

If not, use:

- high
- medium
- low

High confidence:

The PM record has clear field matches and strong property/unit/task/date interpretation.

Medium confidence:

The PM record has usable field matches but missing or ambiguous values.

Low confidence:

The PM record has weak field matches, unknown categories, unresolved assignment, or conflicting values.

Do not auto-upgrade confidence to make reports look better.
