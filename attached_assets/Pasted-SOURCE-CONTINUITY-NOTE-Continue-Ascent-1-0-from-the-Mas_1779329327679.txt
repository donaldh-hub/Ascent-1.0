BUILD 7.3 — NARRATIVE INSIGHTS ENGINE


PRIMARY GOAL

Build the Narrative Insights Engine for Ascent 1.0.

This layer must convert confirmed reporting analysis from Build 7.2 into clear, operational, evidence-backed narrative insights.

The system should not only show numbers.

It should explain:

1. What the data suggests
2. Why it matters operationally
3. What records support the finding
4. What category the issue belongs to
5. Whether the insight is fully supported or partially supported
6. What question the manager should review next
7. What action should be considered next without overprescribing

The output should feel like a practical maintenance operations analyst, not a generic chatbot.

TURN + WORK ORDER INTERPRETATION RULE

Before generating turn-related narrative insights, check the organization’s turn/work order reporting configuration.

Support at least these modes:

1. Separate Mode

Meaning:
Turns and work orders are treated as separate operational categories.

Narrative rule:
Do not imply work order closure equals turn completion.

Example language:
“Work order activity may be contributing to turn progress, but the turn record itself must be reviewed separately.”

2. Work Orders as Turn Progress Mode

Meaning:
The organization uses work order completion inside vacant units as one way to measure turn progress.

Narrative rule:
Narrative insights may explain how open, delayed, or completed work orders are affecting turn progress.

Example language:
“Because this organization tracks turn progress through related work orders, the open work orders in these units are being included as part of turn progress analysis.”

3. Unknown / Not Configured Mode

Meaning:
The organization has not answered the turn/work order interpretation question.

Narrative rule:
Do not make strong turn-progress conclusions from work order data.

Render a setup warning:

“Turn reporting interpretation has not been configured yet. Ascent can show related work order activity, but it will not treat work order closure as turn completion until this setting is confirmed.”


NARRATIVE INSIGHT TYPES

Build the engine so it can generate these practical insight types:

1. Bottleneck Insight

Explains where operational movement appears slowed.

Must include:
- Category
- Affected property
- Affected units or records
- Aging/time evidence
- Count of supporting records
- Whether the bottleneck is confirmed or suggested

Example:
“Maintenance-stage work is creating the clearest turn delay at North Ridge. 8 active turns have remained in maintenance longer than the expected range. Review the related unit records first to confirm whether staffing, parts, or contractor scheduling is driving the delay.”

2. Time Allocation Insight

Explains where labor/time appears concentrated.

Must include:
- Category
- Time pattern
- Comparison point where available
- Supporting records
- Limitation if time data is incomplete

Example:
“Work order time appears concentrated in repeat plumbing repairs. This may be reducing available maintenance capacity for turns and PM work. Review the supporting records to determine whether the issue is isolated to specific units, recurring fixtures, or vendor-dependent repairs.”

3. Risk Insight

Explains where current conditions may create operational risk.

Must include:
- Risk driver
- Evidence
- Operational consequence
- Supporting records
- Review question

Example:
“Open work orders older than the response target are creating service-risk exposure. The issue is concentrated in 12 records across 3 properties. Review whether these delays are caused by parts, staffing, vendor scheduling, or assignment gaps.”

4. Data Quality Insight

Explains when reporting is limited.

Must include:
- Missing fields
- Affected report area
- What can still be reported
- What cannot be trusted yet
- Suggested upload or review step

Example:
“Turn reporting is partially available, but several records are missing vacant-date or stage information. Ascent can show active turn counts, but cycle-time conclusions should remain limited until those fields are completed.”

5. Evidence Gap Insight

Explains missing documentation that weakens proof.

Must include:
- Missing documentation type
- Affected records
- Why documentation matters
- Supporting record list

Example:
“Several high-priority records have no attached documentation. This limits proof for decision-making, especially where delays, contractor work, or resident-facing issues are involved.”

6. Assignment Coverage Insight

Explains where records are unresolved or partially assigned.

Must include:
- Assignment issue
- Count
- Affected records
- Impact on reporting
- Next review question

Example:
“14 uploaded records are only partially assigned. They can support limited trend review, but they should not feed executive certainty until property, unit, or category assignment is confirmed.”


CONFIDENCE AND DATA SUPPORT RULES

Each insight must include a confidence level.

Use:

1. High Confidence

Requirements:
- Based mostly on Fully Reportable records
- Supporting records exist
- Required fields are present
- Counts and filters align

2. Medium Confidence

Requirements:
- Based on mixed Fully Reportable and Partially Reportable records
- Supporting records exist
- Some fields are missing but the pattern is still useful

3. Low Confidence

Requirements:
- Based mainly on partial data
- Insight is directional only
- Must include limitation language

Do not generate strong conclusions from low-confidence data.

Each insight must also include dataSupportLevel:

1. Fully Supported
2. Partially Supported
3. Directional Only
4. Not Enough Data

If dataSupportLevel is “Not Enough Data,” the system should not generate a normal operational insight.

It should generate a reporting readiness message instead.


REPORTABILITY BREAKDOWN

Every narrative insight must show or store its reportability breakdown.

Include:

1. fullyReportableCount
2. partiallyReportableCount
3. notReportableYetCount
4. totalRecordsReviewed
5. percentFullyReportable
6. limitationText

Example limitation language:

“Because 31% of the related records are only partially reportable, this insight should be treated as directional until missing fields are resolved.”

No insight should silently mix full and partial records as if they are equal.


NARRATIVE TONE RULES

Narrative language must be:

1. Plain
2. Operational
3. Specific
4. Evidence-backed
5. Manager-facing
6. Calm
7. Action-oriented
8. Not dramatic
9. Not generic
10. Not fake-consultant language

Avoid phrases like:

- “Unlock powerful insights”
- “Revolutionize your operations”
- “AI has discovered”
- “This proves”
- “Guaranteed”
- “Always”
- “Never” unless it is a system rule

Use language like:

- “The records suggest...”
- “This may indicate...”
- “The strongest signal is...”
- “Review these records first...”
- “This appears concentrated in...”
- “The data is limited because...”
- “Before treating this as confirmed, review...”
- “The next operational question is...”


RECOMMENDED REVIEW QUESTION REQUIREMENT

Every insight must include a recommendedReviewQuestion.

This is one of the most important parts of 7.3.

The insight should not just say what happened.

It should sharpen what the manager should ask next.

Examples:

Work Orders:
“Are these delayed work orders waiting on parts, staffing, vendor scheduling, or assignment correction?”

Turns:
“Are these turn delays being caused by maintenance scope, contractor timing, inspection failure, or unresolved work orders?”

PM:
“Are emergency repairs increasing in categories where PM records are missing or outdated?”

Assets:
“Are repeat repairs happening on assets that are outside warranty or near replacement age?”

Evidence:
“Are high-priority records missing photos, invoices, or completion documentation?”

Assignment:
“Which partially assigned records need property, unit, or category confirmation before they can support reporting?”


SUGGESTED NEXT STEP REQUIREMENT

Every insight must include a suggestedNextStep.

This should not be too bossy.

Do not say:
“Fix this immediately.”

Use:
“Review these records first.”
“Confirm whether parts or vendor delays are the cause.”
“Update missing stage dates before relying on cycle-time reporting.”
“Open the supporting records and confirm whether the pattern is isolated or repeating.”
“Use this record group as the starting point for the next maintenance conversation.”

This is core to Ascent 1.0.

The report brings both sides together.

The Intelligence Engine sharpens the question.

SEVERITY RULES

Use severity levels:

1. Informational
2. Watch
3. Needs Review
4. Priority
5. Critical

Severity must be calculated from existing scoring/risk/time/bottleneck logic where available.

Do not invent a separate severity system if existing scoring and alert logic can support it.

Map severity carefully:

Informational:
Useful observation, low operational pressure.

Watch:
Pattern is forming but not urgent yet.

Needs Review:
Records suggest a meaningful issue that requires manager review.

Priority:
Issue is affecting performance, timing, risk, or capacity.

Critical:
Issue is severe, aging, high-risk, or tied to multiple urgent records.
