## CORE GOAL

Build 7.1 creates the **Reporting Ingestion Layer** for Ascent 1.0.

This layer must collect, normalize, classify, and prepare operational records so the Build 7 reporting spine can generate trustworthy reports from real system data.

The goal is not just to upload files.

The goal is to create a governed reporting intake path that feeds:

1. Reports
2. Analytics
3. Control Tower validation
4. Work order reporting
5. Turn reporting
6. PM reporting readiness
7. Asset reporting readiness
8. Assignment coverage reporting
9. Evidence/documentation reporting
10. Future trend and recalculation engines

The reporting layer must answer:

**What data entered the system?**

**Was it connected correctly?**

**Can it safely be used for reporting?**

**What still needs review?**

That is the difference between a dashboard that looks right and a system that can prove it.

---

## CRITICAL SYSTEM PRINCIPLE

All imported or ingested data must follow:

**SOURCE FILE → NORMALIZED RECORD → GOVERNANCE STATUS → ASSIGNMENT STATUS → REPORTING ELIGIBILITY → SUPPORTING RECORDS**

No report should treat a record as fully reportable unless the record is properly classified.

No dashboard or Control Tower output should silently mix fully resolved and partially resolved records without saying so.

No reporting metric should imply certainty where the source data is incomplete.

# 4. REPORTING ELIGIBILITY CLASSIFICATION

Every normalized record must receive a reporting eligibility classification.

Required classifications:

## A. FULLY REPORTABLE

Use when:

1. Organization scope is valid
2. Property link is valid where required
3. Unit link is valid where required
4. Assignment confidence is high or confirmed
5. Resolution status is fully_resolved
6. Source record has enough data to support the report family

Behavior:

1. Can feed reports
2. Can feed analytics
3. Can feed Control Tower validation
4. Can appear in drill-downs as confirmed truth
5. Can contribute to unit-level and property-level reporting

## B. PARTIALLY REPORTABLE

Use when:

1. Organization scope is valid
2. Property may be valid
3. Unit or asset context is incomplete
4. Resolution status is partially_resolved
5. Record can support limited rollups but not precise unit-level claims

Behavior:

1. Can feed clearly labeled property-level or intake-quality reports
2. Can appear in reporting readiness summaries
3. Must be excluded from confirmed unit-level truth
4. Must not silently inflate confirmed operational counts
5. Drill-down must label limitations

## C. NOT REPORTABLE YET

Use when:

1. Property match failed
2. Organization scope is unclear
3. Required linkage is missing
4. Assignment confidence is low
5. Resolution status is unresolved
6. Record is awaiting manual review

Behavior:

1. Must not feed executive reporting truth
2. Must not feed confirmed Control Tower performance metrics
3. Must appear in data quality / review reports
4. Must remain available for manual resolution
5. Must preserve source data

5. Build 7.1 through 7.9 testing
