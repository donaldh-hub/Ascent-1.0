BUILD PROMPT (COPY + PASTE INTO REPLIT AGENT)
=============================================

We are continuing the phased build of Ascent 1.0.

This is PHASE 1 — BUILD 1.6.

The system already has:

* authentication
* organization structure
* user roles
* protected routes
* app shell
* settings foundation
* workflow engine
* document + evidence engine
* evidence visibility layer
* signal strength + language layer
* guided system activation + unit intake layer
* units page foundation

Do NOT rebuild the foundation.
Do NOT redesign the application.
Do NOT create a second data model for unit history.
Do NOT create fake work history or placeholder-only insight text.
Do NOT create unit pages that are visually separate from real system records.

This build is the UNIT PROFILE + IMMEDIATE HISTORY VIEW.

PROJECT NAME:
Ascent 1.0

CORE GOAL OF THIS BUILD:

Create a real unit-centered profile page that allows the user to click a unit and immediately see the work, documents, assets, and history already connected to that unit.

This build must make the system feel real immediately after onboarding.

The user should be able to say:

“I clicked the unit… and now I can actually see what has happened there.”

This build is NOT advanced intelligence yet.
This build is immediate visibility, context, and trust.

==================================================

CRITICAL SYSTEM RULE

==================================================

This build MUST read from the existing shared system structure.

That means:

* unit records must come from Build 1.5 data
* workflow history shown here must come from real workflow/item records if linked
* documents shown here must come from the existing document/evidence engine
* evidence badges/signals must reuse the existing evidence visibility + language layer
* no detached unit-profile-only data objects
* no shadow copies of documents or work history

This build must strengthen the SINGLE SOURCE OF TRUTH.

==================================================

BUILD REQUIREMENTS

==================================================

1. UNIT LIST PAGE UPGRADE

Upgrade the /units page from Build 1.5.

Requirements:

* list all units for the organization
* search units
* filter by property if cleanly supported
* each unit row/card must be clickable
* click opens dedicated unit detail route

Route:

* /units/:unitId

Each unit row/card should show at minimum:

* unit name / number
* property name
* created date
* document/evidence signal if available
* simple work/history count if available

Do NOT clutter the units page.
Keep it clean and operational.

==================================================

2. UNIT PROFILE PAGE

Build a dedicated Unit Profile page.

The page must show the selected unit as a real operational record.

Required top section:

A. Unit Header

* unit name / number
* property name
* created date
* optional metadata if available

B. Quick Snapshot
Show simple real counts only if they exist:

* related work items count
* related documents count
* related assets count
* latest activity date

If counts do not exist yet, show truthful empty or low-data states.

==================================================

3. UNIT HISTORY TIMELINE

Build an immediate history section for the unit.

This is a key requirement.

The timeline must surface real history connected to the unit where available.

Acceptable timeline event types:

* workflow item created for this unit
* workflow item moved stages for this unit
* document uploaded and linked to this unit
* document attached to a unit-linked workflow item
* asset created and linked to this unit
* manual unit setup/import event if little else exists yet

Each event should show:

* event type
* short title
* timestamp
* linked record reference if appropriate
* actor or system source if available

Behavior requirements:

* newest first
* clear empty state if no activity exists yet
* no fake historical summaries

This timeline should help the user immediately understand what has happened in the unit.

==================================================

4. UNIT DOCUMENTS / EVIDENCE SECTION

Add a dedicated documents/evidence section to the Unit Profile.

This section must read from the existing document engine.

Requirements:

* show documents directly linked to the unit if supported
* also show documents linked to workflow items associated with the unit if available
* reuse existing evidence signals and language patterns
* show attachment count
* allow click to preview using existing preview behavior
* do NOT force the user to leave the unit context just to understand the evidence

Display at minimum:

* file name
* document type if tagged
* upload timestamp
* uploader if available
* evidence state label using shared language system

If no documents exist:

* show clear empty state
* do not imply missing documentation unless a real rule from existing layers applies

==================================================

5. UNIT WORK / ISSUE SECTION

Build a unit-level work section.

This section should aggregate real workflow items associated with the unit.

Requirements:

* show related workflow items if they exist
* show workflow name
* item title
* priority
* current stage
* current status if available
* updated date

User must be able to click into the workflow item or related workflow from the unit page.

If no workflow items are linked yet:

* show truthful empty state
* do not fabricate issue summaries

This section is not advanced issue intelligence yet.
This is a visibility layer over real linked work.

==================================================

6. UNIT ASSETS SECTION

Add a unit assets section.

This section should show assets linked to the unit if assets already exist in the system.

Requirements:

* list linked assets
* asset name or type
* manufacturer if available
* model/serial if available
* status if available
* created or install date if available

No advanced lifecycle intelligence yet.
No predictive maintenance logic yet.

If no assets exist:

* show empty state

==================================================

7. SIMPLE UNIT INSIGHT STRIP

Add a lightweight insight strip near the top of the Unit Profile.

This must use ONLY real currently available data.

Examples of acceptable insight types:

* “3 related work items recorded for this unit.”
* “2 documents are attached to unit-related activity.”
* “No work history has been recorded yet.”
* “Latest activity was a document upload.”

Do NOT create advanced pattern detection.
Do NOT imply recurring issues unless there is real repeated linked data.
Do NOT invent summary language.

==================================================

8. DATA LINKING REQUIREMENTS

This build must support and respect shared linking fields.

Where cleanly needed, ensure records can relate through:

* use what is real now
